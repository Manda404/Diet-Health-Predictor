# Diet Health Predictor 🍎📊

A clean architecture data science package for predicting diet and health outcomes.

## Overview

This package demonstrates best practices for building data science applications using **Clean Architecture** principles. It includes:

- ✅ Multi-environment configuration (dev, staging, prod)
- ✅ YAML-based settings management
- ✅ Clean Architecture layers (Domain, Application, Infrastructure, Presentation)
- ✅ Proper separation of concerns
- ✅ Type hints and validation with Pydantic
- ✅ Comprehensive logging
- ✅ Poetry for dependency management
- ✅ Data cleaning, feature engineering, and encoding/scaling (scikit-learn)
- ✅ Boosting model training, cross-validation, and comparison (XGBoost, CatBoost)

## Quick Start

### Prerequisites
- Python 3.14+
- Poetry

### Installation

1. **Clone/setup the project:**
```bash
cd Diet-Health-Predictor
```

2. **Install dependencies:**
```bash
poetry install
```

3. **Activate the virtual environment:**
```bash
poetry shell
```

4. **Set environment (optional):**
```bash
# Create .env file
cp .env.example .env

# Edit .env to set ENVIRONMENT=development (default)
```

### Running

**Get a summary of the dataset:**
```bash
python -m diet_health_predictor.presentation
```

**Using in Python:**
```python
from diet_health_predictor.presentation import HealthDietAPI

# Initialize API
api = HealthDietAPI()

# Load data
assessments = api.load_data()

# Get statistics
stats = api.get_health_statistics(assessments)
print(stats)

# Run the Phase 2 pipeline: clean, engineer features, split, encode/scale, persist
result = api.preprocess_data()
print(result.X_train.shape, result.feature_names)

# Run the Phase 3 pipeline: train a boosting model, evaluate, persist
from diet_health_predictor.application import ModelType

training_result = api.train_model(ModelType.XGBOOST, result)
print(training_result.metrics["accuracy"])
```

## Configuration Management

### Environment Variables

The application supports three environments:

| Environment | Purpose | Use Case |
|------------|---------|----------|
| **development** | Local development | Feature development, debugging |
| **staging** | Pre-production testing | Integration tests, performance testing |
| **production** | Live deployment | Production inference, real data |

### Setting Environment

**Option 1: Environment Variable**
```bash
export ENVIRONMENT=development
python -m diet_health_predictor.presentation
```

**Option 2: .env File**
```bash
# Create .env file in project root
ENVIRONMENT=development
LOG_LEVEL=DEBUG
```

**Option 3: Programmatically**
```python
import os
os.environ['ENVIRONMENT'] = 'staging'

from diet_health_predictor.config import Settings
settings = Settings.from_yaml(env='staging')
```

### Configuration Files

Each environment has a YAML configuration file:

**`config/settings.dev.yaml`** - Development (verbose logging, all data)
**`config/settings.staging.yaml`** - Staging (normal logging, full data)
**`config/settings.prod.yaml`** - Production (minimal logging, optimized paths)

Example configuration structure:
```yaml
environment: development
debug: true
log_level: DEBUG

data:
  raw_data_path: "data/healthy_diet_calorie_intake.csv"
  processed_data_path: "data/processed"
  sample_size: null

model:
  test_size: 0.2
  random_state: 42
  models_output_dir: "models"
  # Metric select_best_model() uses to pick a winner from compare_models().
  selection_metric: "mcc"
  # Passed directly to XGBClassifier(**xgboost_params) / CatBoostClassifier(**catboost_params).
  # This is the only source of model hyperparameter defaults -- the wrapper
  # code hardcodes nothing but random_state. Omit a key (or the whole
  # section) to fall back to that library's own default instead.
  xgboost_params:
    n_estimators: 200
    max_depth: 6
    learning_rate: 0.1
    eval_metric: "mlogloss"
  catboost_params:
    iterations: 200
    depth: 6
    learning_rate: 0.1
    verbose: false
```

## Architecture Layers

### 🏛️ Clean Architecture Overview

```
┌─────────────────────────────────────────┐
│       PRESENTATION LAYER (API)          │
│  HealthDietAPI, Controllers, Routes     │
└──────────────────┬──────────────────────┘
                   │ depends on
┌──────────────────▼──────────────────────┐
│      APPLICATION LAYER (Use Cases)      │
│  LoadData, AnalyzeStats, Orchestration  │
└──────────────────┬──────────────────────┘
                   │ depends on
┌──────────────────▼──────────────────────┐
│        DOMAIN LAYER (Business Logic)    │
│  Person, DietAssessment, HealthStatus   │
└──────────────────┬──────────────────────┘
                   │ depends on
┌──────────────────▼──────────────────────┐
│    INFRASTRUCTURE LAYER (I/O, DB)       │
│  DataLoader, FileOperations, API Calls  │
└─────────────────────────────────────────┘
```

### Layer Details

#### 1. **Domain Layer** (`domain/`)
- **Purpose:** Core business logic independent of any framework
- **Contains:** Entities, Value Objects, Business Rules
- **Example:** `Person`, `DietAssessment`, `HealthStatus`
- **Key Principle:** No external dependencies

#### 2. **Application Layer** (`application/`)
- **Purpose:** Orchestrate use cases and business logic
- **Contains:** Use Cases, Service Orchestration
- **Example:** `LoadHealthDietDataUseCase`, `AnalyzeHealthStatsUseCase`
- **Key Principle:** Depends on Domain, coordinates Infrastructure

#### 3. **Infrastructure Layer** (`infrastructure/`)
- **Purpose:** Handle external I/O operations
- **Contains:** Data Loaders, Database Access, External APIs
- **Example:** `HealthDietDataLoader`, Data validation
- **Key Principle:** Adapts external systems to domain models

#### 4. **Presentation Layer** (`presentation/`)
- **Purpose:** Expose functionality to the outside world
- **Contains:** APIs, CLI, Web interfaces
- **Example:** `HealthDietAPI`, main entry points
- **Key Principle:** Depends on Application layer

#### 5. **Configuration Layer** (`config/`)
- **Purpose:** Manage environment-specific settings
- **Contains:** Settings models, YAML loaders
- **Key Principle:** Centralized configuration management

## Data Preprocessing & Feature Engineering (Phase 2)

`HealthDietAPI.preprocess_data()` runs the full pipeline and writes its output to
`data/<processed_data_path>/`:

1. **Deduplicate** (`DataCleaner.drop_duplicates()`) - runs on the full dataset,
   *before* splitting (an exact-duplicate row must not end up once in train and once in test)
2. **Split** (`DataSplitter`) - stratified train/test split (`model.test_size` /
   `model.random_state` from settings)
3. **Clean** (`DataCleaner.fit()` / `.transform()`) - imputes missing values (median/mode)
   and clips numeric outliers (IQR method); fit on the training split only, then applied
   to test, so test-set statistics never leak into the values used to clean training data
4. **Engineer features** (`FeatureEngineer`) - derives 14 columns:
   `Age_Group`, `Calorie_Balance`, `Calorie_Deviation_Pct`, `Protein_Ratio`, `Carb_Ratio`,
   `Fat_Ratio`, `Total_Macros_g`, `Adequate_Water_Intake`, `BMR` (Mifflin-St Jeor),
   `Activity_Calorie_Multiplier` (requirement / BMR), `Ideal_Weight_kg` / `Weight_Deviation_kg`
   (Devine formula), `Protein_per_kg_Bodyweight`, `Water_Intake_ml_per_kg`
5. **Encode & scale** (`FeatureTransformer`) - `StandardScaler` for numeric columns,
   `OneHotEncoder` for categorical columns, fit on train only
6. **Persist** (`ProcessedDataWriter`) - `X_train.csv`, `X_test.csv`, `y_train.csv`,
   `y_test.csv`, and `feature_transformer.joblib` (reusable for inference)

See `notebooks/02_data_preprocessing.ipynb` for a walkthrough of each step.

> **Data leakage:** `Health_Status` is an *exact* deterministic function of `BMI`
> in this dataset (WHO BMI thresholds, verified with zero exceptions across all
> 6000 rows) — keeping `BMI` as a feature gives a trivial 100% accuracy. `Height_cm`/
> `Weight_kg` reconstruct `BMI` almost perfectly, and several engineered columns
> (`BMR`, `Ideal_Weight_kg`, `Weight_Deviation_kg`, the two per-kg-bodyweight ratios)
> are themselves derived from `Height_cm`/`Weight_kg`. All of these are still
> computed by `FeatureEngineer` (useful for analysis) but excluded from the model's
> feature set via `LEAKING_NUMERIC_COLUMNS` in `feature_engineering.py`. Full
> investigation, with before/after accuracy numbers, in
> `notebooks/03_model_training.ipynb`, section 1.

## Model Training (Phase 3)

Two boosting classifiers are supported, each behind the same `fit`/`predict`/
`save`/`load`/`get_evals_result` contract (`infrastructure/models/`):
`XGBoostWrapper`, `CatBoostWrapper`. All of their hyperparameters come from
`settings.model.{xgboost,catboost}_params` (YAML) — the wrapper code hardcodes
nothing but `random_state`; an unset key falls back to that library's own
built-in default.

```python
from diet_health_predictor.presentation import HealthDietAPI
from diet_health_predictor.application import ModelType

api = HealthDietAPI()
prep = api.preprocess_data()

# Train + evaluate + persist a single model -- hyperparameters from YAML...
result = api.train_model(ModelType.XGBOOST, prep)
# ...or pass them explicitly for this call only, ignoring the YAML section:
# result = api.train_model(ModelType.XGBOOST, prep, hyperparameters={"n_estimators": 300})
print(result.metrics)          # accuracy, precision/recall/f1 (macro), mcc, confusion matrix
print(result.evals_result)     # per-iteration train vs. validation curve

# Stratified k-fold cross-validation (no persistence -- a stability check)
cv = api.cross_validate_model(ModelType.XGBOOST, prep, n_splits=5)
print(cv.mean_metrics, cv.std_metrics)

# Train every registered model on the same split, then pick + persist a winner
comparison = api.compare_models(prep)
winner = api.select_best_model(comparison)  # by settings.model.selection_metric ("mcc")
```

Each model's artifacts are written to `settings.model.models_output_dir/<model_type>/`:
`model.joblib` and `label_encoder.joblib`. `select_best_model()` additionally
copies the winner's artifacts to `settings.model.models_output_dir/best/`
(plus a `metadata.json` recording which model won and by what metric/value) —
a canonical location a downstream consumer can load "the" model from without
knowing which `ModelType` won. The selection metric defaults to **MCC**
(Matthews Correlation Coefficient, `settings.model.selection_metric`) rather
than accuracy or f1 macro, since it stays meaningful under class imbalance;
override it per call with `select_best_model(comparison, metric="f1_macro")`,
or use `best_model(comparison, metric=...)` directly if you just want the
winner without persisting anything.

See `notebooks/03_model_training.ipynb` for a full walkthrough, including
plotting the train/eval curves.

## Continuous Integration

`.github/workflows/ci.yml` runs on every push and pull request against `main`:

1. `poetry check --lock` - fails fast if `poetry.lock` drifts from `pyproject.toml`
2. `black --check`, `isort --check-only`, `flake8` - formatting/lint
3. `mypy src` - type-checking
4. `pytest tests/unit -m unit` and `pytest tests/integration -m integration` - the full test suite, run as two separate steps so a unit failure and an integration failure are never conflated in the log
5. Coverage XML reports uploaded as build artifacts

`poetry.lock` is committed (not gitignored) specifically so CI installs the
exact dependency versions every contributor uses locally.

## Logging

The application automatically configures logging based on environment:

- **Development:** DEBUG level, console + file output
- **Staging:** INFO level, console + file output
- **Production:** WARNING level, file output only

Logs are stored in `logs/{environment}.log`
